# TaskOrbit — backend

Python orchestration layer for the TaskOrbit voice agent. The backend
module layout under `src/taskorbit/` mirrors the architecture document
exactly:

| Module | Status | Role |
|---|---|---|
| `api/` | implemented | FastAPI HTTP layer (health endpoint today) |
| `livekit_agent/` | stub | LiveKit agent worker — joins rooms, runs the voice pipeline |
| `orchestration/` | stub | Routes the conversation, plans workflows, dispatches tools |
| `agents/` | stub | Task-specific agents (technical, sales, …) |
| `tools/` | stub | Tool implementations exposed to the LLM |
| `integrations/` | stub | Adapters to Meisterwerk and other external APIs |
| `language/` | stub | Multilingual handling |
| `logging/` | implemented | Structured logging configuration (structlog) |
| `database/` | stub | PostgreSQL access layer (sqlalchemy/alembic deps declared) |

The "stub" modules each contain a single `__init__.py` with a docstring
describing what goes there. Their feature implementations are owned by
downstream tickets.

---

## Setup

```bash
cd backend
cp .env.example .env          # then fill in API keys
poetry install
```

`poetry install` reads `pyproject.toml` + `poetry.lock` and creates a
virtualenv. With `poetry config virtualenvs.in-project true` (recommended)
the venv lives at `backend/.venv/`.

## Smoke test

After `poetry install`:

```bash
poetry run pytest             # runs the test suite
poetry run ruff check .       # linting
poetry run taskorbit-api      # start API
curl http://localhost:8000/health
# → {"status":"ok","service":"taskorbit-backend","version":"0.1.0"}

# LiveKit token endpoint (ticket #26). Requires LIVEKIT_* vars in .env.
curl -X POST http://localhost:8000/v1/livekit/token \
  -H "Content-Type: application/json" \
  -d '{"identity":"dev-user","room":"taskorbit-dev-room"}'
# → {"token":"...","url":"wss://...","room":"...","identity":"..."}
```

See [`docs/livekit-cloud-setup.md`](../docs/livekit-cloud-setup.md) for
the full LiveKit Cloud developer setup.

## A note on the `logging/` module name

`taskorbit.logging` shadows the stdlib `logging` package in the
absolute-import sense, but Python 3's import system handles this
cleanly: `import logging` from anywhere outside the package still
gets the stdlib. Inside the `taskorbit/logging/` directory, prefer
absolute imports (`import logging`) when you mean the stdlib, and
`from taskorbit.logging.setup import ...` when you mean our helpers.

This naming is intentional — it matches the module list in the
architecture document.
